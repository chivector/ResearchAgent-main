# **上下文是一个科学控制变量**



长上下文并不只是效率负担；在潜在评分标准的科学推理中，它是一种未受控制的认知干预。既有研究表明，LLM 并不会均匀地使用长上下文。“Lost in the Middle” 证明了长上下文使用中的位置依赖性退化，即模型往往会低度使用放在提示开头或结尾之外的证据。LongBench 进一步表明，长上下文理解仍然是一种独立且困难的能力，并不会仅仅因为扩大上下文窗口而得到解决。LLMLingua 和 LongLLMLingua 等提示压缩工作表明，减少上下文可以通过提高关键信息密度来保持甚至提升性能。近期的智能体记忆工作，包括 MemGPT、MEM1 和 Memory-as-Action，则将该领域推向显式记忆管理，而不是把所有内容都追加进提示中。

对于我们的领域而言，问题更加尖锐。FrontierScience-Research 并不是普通的长上下文问答：答案是开放形式的、按过程评分的，并且依赖潜在评分标准。**一个载荷不仅必须保留事实，还必须保留与评分相关的义务**：精确方程、约束、单位、排除项、控制条件、中间辅因子计数、端点假设以及特定领域惯例。因此，无关上下文在两个方向上都很危险：它既可能掩埋一个必要的评分标准原子，也可能激活一个貌似合理但越界的科学框架。这正是我们已经观察到的失败模式：物理智能体接收到化学/生物残留信息，或者代谢计算从题目的计数边界漂移到一个通常合理的完全氧化模型中。



### (1) **形式化问题定义**

设第 $t$ 阶段后的全局记忆状态为：

$$
M_t=\{m_1,\dots,m_n\}
$$

其中每个记忆原子 $m_i$ 都包含类型、来源、成本、领域、支持状态以及指向提示词义务的链接。对于角色为 $z_a$、局部任务为 $q_a$、契约为 $K_a$、义务集为 $O_a$ 的子智能体 $a$，我们必须构建一个载荷：

$$
C_a \subseteq M_t
$$


理想的载荷既不是最大的上下文，也不是最短的摘要。它是**最小充分科学上下文**：

$$
C_a^\star=\arg\min_{C\subseteq M_t}
\mathbb{E}[L_a(C)] + \lambda |C| + \mu D_a(C)
$$


其中 $L_a(C)$ 是期望任务损失，$|C|$ 是令牌成本，$D_a(C)$ 是干扰项风险：即不相关原子诱发非法承诺、超出范围的推理或跨领域污染的概率。

如果满足以下条件，则一个载荷是 $\epsilon$-充分的：

$$
R_a(C)-R_a(M_t)\leq \epsilon
$$

其中 $R_a$ 是相对于完整记忆的期望局部风险。如果没有更低成本的载荷能满足相同的边界，则该载荷是最小的。这为我们提供了一个清晰的理论目标：**在保持义务覆盖和支持有效推理的同时，最小化上下文**。



### (2) **理论基础**

最优雅的基础是**义务条件下的信息瓶颈**。经典信息瓶颈理论要求找到一个压缩表示，使其保留关于目标变量的信息。在这里，目标不是类别标签，而是成功完成一个科学契约：

$$
\min I(C;M_t)-\beta I(C;Y_a \mid O_a,K_a)
$$


其中 $Y_a$ 是正确的局部产物：推导模块、机理图、比较矩阵、协议对象或主张许可决策。我们可以通过引入干扰信息惩罚来扩展该目标：

$$
\min I(C;M_t)-\beta I(C;Y_a\mid O_a,K_a)+\gamma I(C;N_a)
$$


其中 $N_a$ 表示干扰变量：不相关的领域字段、未经支持的假设、过时的中间输出或过度泛化的科学先验。

第二个基础是受约束的次模优化。如果每个记忆原子对义务覆盖和证据支持的边际贡献是递减的，我们可以定义：

$$
U_a(C)=
\alpha\,\mathrm{Coverage}(C,O_a)
+\beta\,\mathrm{Support}(C)
+\rho\,\mathrm{Dependency}(C)
-\gamma\,\mathrm{Noise}(C)
$$

约束条件为 $|C|\leq B_a$ 。这提供了一种原则性的方式来讨论近似、消融和载荷质量，而不是将上下文过滤呈现为手工制定的规则。



### (3) **理想解决方案**

理想的方法不是“按智能体类型过滤上下文”。这仅仅是一个 Baseline。可发表的版本应当是：

> **义务条件化认知载荷优化**：一个可学习或可优化的上下文选择层，为每次子智能体调用构造角色特定、保留来源、最小充分的载荷。

该方案应满足六个要求：

1. **义务完整性**：与当前子任务相关的每个高优先级提示义务都必须被表示。
2. **支持充分性**：每个可断言的声明都必须具有可访问的证据、推导或领域标准依据。
3. **干扰抑制**：无关记忆原子应被主动惩罚，而不只是通过启发式过滤被省略。
4. **角色自适应性**：推导智能体、机制智能体、协议智能体和写作智能体的载荷应在结构上不同。
5. **可追踪来源**：每个被纳入的原子都应说明其被纳入的原因，以及它支持哪项义务或声明。
6. **动态可修订性**：如果下游批评器检测到遗漏或幻觉，载荷策略应更新记忆相关性，而不是只修补文本。

### 技术路线示例（仅供参考，可调整）



理想设计是一种**双重记忆架构**：

```text
规范记忆图
  细粒度类型化原子、链接、来源、支持状态

物化认知视图
  从图中编译出的角色特定、任务特定、预算感知载荷
```

**为什么仅有原子会失败**
原子记忆处理的是“我应该检索哪个事实？”，但无法处理“我处于什么推理状态中？” FrontierScience 风格的任务两者都需要。

原子至少会在五个方面失败：

1. **组合依赖丢失**：公式、变量定义、边界条件和近似可能只有作为一个推导束时才有意义。
2. **全局不变量丢失**：诸如“终止于草酰乙酸”或“不要直接使用 1.4-1.3 ppm 处的峰”等约束，必须治理后续许多原子。
3. **评分标准形状丢失**：评分通常奖励成组结构：控制条件 + 读出指标 + 参数，或方程 + 变换 + 最终表达式。
4. **冲突盲区**：两个原子单独看都可能合理，但合在一起却不一致。
5. **生命周期盲区**：有些记忆在批评器标记遗漏后应变得更重要；另一些记忆则应衰减或被抑制。

所以，需要多层级记忆，但不是静态字典式的多层级。需要的是**类型化、可演化、契约条件化的记忆视图**。

**理想记忆模型**
一个强有力的学术设计可以称为：

> **义务条件化层级记忆图**

每个记忆原子 $m_i$ 都具有元数据：

$$
m_i=(c_i,\tau_i,p_i,s_i,o_i,d_i,r_i,\kappa_i)
$$


其中 $c_i$ 是内容，$\tau_i$ 是类型，$p_i$ 是来源，$s_i$ 是支持状态，$o_i$ 是链接的义务，$d_i$ 是依赖项，$r_i$ 是角色相关性，$\kappa_i$ 是 token 成本。

随后，每次智能体调用都会构造一个载荷：

$$
C_a^\star=\arg\max_{C\subseteq M}
U(C; a, q, O, K)-\lambda \mathrm{Cost}(C)-\gamma \mathrm{DistractorRisk}(C)
$$


其中 $a$ 是智能体角色，$q$ 是任务，$O$ 是提示义务集合，$K$ 是当前契约。

**推荐层级**
不是之前静态的 `_core/_background/_entities/_meta`，而应类似于：

```text
第 0 层：任务契约
问题意图、当前子任务、必需输出形状、禁止扩展项

第 1 层：问题内核
不可约事实、变量、端点、边界条件、精确提示约束

第 2 层：类型化工作记忆
证据原子、声明原子、推导节点、协议节点、比较关系

第 3 层：结构状态
任务图、义务图、声明图、依赖图、支持-许可格

第 4 层：情节轨迹
智能体输出、失败尝试、批评器发现、补丁历史

第 5 层：完整归档
完整问题文本、原始输出、完整检索证据、日志
```

关键创新在于，层级**不是存储文件夹**。它们是**同一个底层记忆图之上的编译视图**。



**交付物**

它应包含：

1. **理论部分**  
   形式化定义记忆问题：原子记忆、分层视图、载荷充分性、干扰风险、义务覆盖和支持保留。
2. **模式规范**  
   定义记忆原子类型：`ProblemFact`、`ConstraintAtom`、`PromptObligationAtom`、`EvidenceAtom`、`ClaimAtom`、`DerivationNode`、`ProtocolNode`、`CriticFinding`、`PayloadView`。
3. **层级/视图设计**  
   解释每一层、其包含内容、何时更新，以及哪些智能体会消费它。
4. **载荷编译器设计**  
   说明一次智能体调用如何获得定制化载荷。包括评分因素：角色相关性、义务链接、支持状态、依赖中心性、token 成本和干扰风险。
5. **生命周期规则**  
   定义提升、抑制、衰减、冲突标记、批评器触发的重新加权，以及来源保留。
7. **评估计划**  
   提出消融实验的设计：完整上下文 vs 静态过滤 vs 原子检索 vs 层级载荷编译。指标包括：token 成本、义务覆盖率、无支持声明率、领域污染率、最终评分标准得分。

## 我们的方案：义务条件化认知载荷编译

我们建议将该方向收敛为一个明确的方法：**义务条件化层级记忆图**与**义务感知载荷编译器**。英文可命名为：

```text
OC-HMG: Obligation-Conditioned Hierarchical Memory Graph
OPC: Obligation-aware Payload Compiler
```

核心观点是：ResearchAgent 的上下文管理不应被定义为“如何减少 prompt token”，而应定义为“如何为每一次子智能体调用构造最小充分、可追踪、低干扰的科学认知载荷”。也就是说，我们不把上下文视为历史记录，而把它视为一个会改变推理路径的控制变量。

### 1. 总体目标

给定全局记忆状态 $M_t$、当前子任务 $q_a$、智能体角色 $a$、任务契约 $K_a$ 和义务集合 $O_a$，系统不再把完整 dossier 或静态摘要直接交给子智能体，而是先经过一个载荷编译步骤，生成：

$$
C_a = \mathrm{Compile}(M_t, a, q_a, O_a, K_a, B_a)
$$

其中 $B_a$ 是 token 或注意力预算。理想载荷必须同时满足四个条件：

1. **义务覆盖**：当前任务必须回答的评分点、格式要求、边界条件和禁止扩展项都被显式保留。
2. **支持闭包**：每个允许写入答案的声明，都能追溯到题面事实、证据片段、推导节点或领域标准。
3. **依赖闭包**：公式、变量定义、单位、端点假设、控制条件和前置推导不能被拆散。
4. **干扰抑制**：无关领域、过期中间结论、未经支持的假设和越界科学框架要被降权或排除。

因此，我们优化的不是“上下文越短越好”，而是：

$$
\max_C
\alpha\,\mathrm{Coverage}(C,O_a)
+\beta\,\mathrm{SupportClosure}(C)
+\rho\,\mathrm{DependencyClosure}(C)
+\eta\,\mathrm{RoleFit}(C,a)
-\gamma\,\mathrm{DistractorRisk}(C)
-\lambda\,\mathrm{Cost}(C)
$$

### 2. 系统架构

整体流程如下：

```text
Problem Text
  -> Context / Intent / Constraint / Axiom extraction
  -> Obligation Graph
  -> Typed Memory Graph
       ProblemFact / ConstraintAtom / PromptObligationAtom
       EvidenceAtom / ClaimAtom / DerivationNode / ProtocolNode
       CriticFinding / PayloadView
  -> Payload Compiler
       role + task + obligations + budget + dependency closure
  -> Sub-agent Call
  -> Output Parser
  -> Verification / Critic / Contract Check
  -> Memory Lifecycle Update
  -> next Payload Compiler call
```

其中，**Typed Memory Graph** 是底层统一记忆状态；**PayloadView** 是每次调用时从图上编译出的角色特定视图。层级不是文件夹，也不是固定摘要，而是同一张记忆图上的不同编译视图。

### 3. 记忆原子设计

每个记忆原子建议具有如下字段：

```text
MemoryAtom:
  id: 全局唯一 ID
  atom_type: problem_fact | constraint | obligation | evidence |
             claim | derivation | protocol | critic_finding
  content: 原子内容
  source: 来源，例如 problem_text、context_miner、formal_deriver、critic
  source_span: 可选，指向原题或中间输出中的片段
  domain: physics | chemistry | biology | mixed | unknown
  support_license: prohibited | unsupported | speculative |
                   plausible | supported | verified
  obligations: 该原子服务的义务 ID
  dependencies: 必须共同出现的依赖原子 ID
  conflicts: 与该原子冲突的原子 ID
  role_affinity: 不同角色的相关性权重
  token_cost: 估计 token 成本
  priority: 当前生命周期优先级
  distractor_risk: 干扰风险
  lifecycle: active | stale | suppressed | archived
```

这里最重要的是 `support_license`。它决定某个内容在系统中的使用权限：

```text
prohibited < unsupported < speculative < plausible < supported < verified
```

不同智能体的许可规则不同：

- 假设建模智能体可以接触 `speculative`，但必须保留其不确定性和被排除的备选解释。
- 形式推导智能体只能使用 `supported` 或 `verified` 的硬条件、变量定义和边界条件。
- 写作智能体默认只能把 `supported` 或 `verified` 内容作为主张写入最终答案。
- 批评器和修补器优先消费 `critic_finding`、`violated constraint` 和 `unsupported claim`。

这可以防止一个“看似合理但未被题面支持”的科学框架越权进入最终答案。

### 4. 层级视图

我们建议保留六层视图，但将它们定义为编译视图，而非存储目录：

```text
L0 Task Contract View
  当前子任务、输出形状、必须回答项、禁止扩展项

L1 Problem Kernel View
  题面显式事实、变量、单位、端点、边界条件、negative constraints

L2 Evidence / Claim Work View
  证据原子、候选声明、支持状态、rejected alternatives

L3 Structural Reasoning View
  task graph、obligation graph、claim-support graph、derivation dependency graph

L4 Critic Trajectory View
  verification issues、proof gaps、patch history、失败尝试

L5 Archive View
  完整原题、完整检索证据、原始中间输出、日志
```

不同角色消费不同组合：

- `TaskGraphBuilder`：主要消费 L0、L1、L3。
- `AxiomBuilder`：消费 L1，且不能把 soft interpretation 升级成 hard axiom。
- `HypothesisModeler`：消费 L1、L2，但必须输出支持等级。
- `FormalDeriver`：消费 L0、L1、L3，重点是公式、变量、单位、边界条件和推导依赖。
- `Verification`：消费 L0-L4，用于查义务遗漏、支持断裂和越界声明。
- `Writer`：消费 L0-L3 中 `supported/verified` 的内容，加少量 L4 风险提醒。
- `Patcher`：消费 L4 和受影响的局部子图，而不是完整历史。

### 5. 载荷编译流程

一次子智能体调用前，OPC 执行以下步骤：

1. **确定当前义务集**  
   从题目意图、constraint ledger、task deliverable、micro-checklist 和上游 critic finding 中提取当前必须覆盖的义务。

2. **生成候选原子集**  
   从问题内核、证据、推导、任务输出、约束账本、批评器反馈中收集可能相关的原子。

3. **强制加入硬约束**  
   当前任务的输出形状、禁止扩展项、显式边界条件、单位、变量定义、negative constraints 必须进入载荷。

4. **做支持闭包和依赖闭包**  
   如果选择某个 claim，就必须包含它的 evidence 或 derivation support；如果选择某个 derivation，就必须包含变量映射、前置公式和边界条件。

5. **计算原子效用分数**  
   使用义务链接、支持等级、角色相关性、依赖中心性、critic 提升、token 成本、干扰风险和生命周期状态进行打分。

6. **预算内选择**  
   先加入强制原子，再用 greedy 或 beam search 选择剩余原子。对公式束、协议束、控制条件组等组合依赖，按 bundle 选择，而不是拆成孤立原子。

7. **渲染 PayloadView**  
   按稳定顺序输出：任务契约、必须约束、问题内核、证据/推导、允许声明、风险提醒、依赖输出。高优先级义务放在开头或结尾，降低 Lost-in-the-Middle 风险。

8. **记录可解释日志**  
   每个 PayloadView 记录被选择的原子、被省略的高分原子、覆盖报告、支持报告、风险报告和 token 成本，供后续消融和错误分析。

### 6. 生命周期更新规则

批评器不应只修补最终文本，还应更新记忆图。建议规则如下：

```text
missing obligation
  -> 提升对应 obligation / evidence / constraint 原子的 priority
  -> 新增 critic_finding 原子

unsupported claim
  -> 将对应 claim 降级为 unsupported 或 prohibited
  -> 下次 writer payload 强制要求 support closure

cross-domain contamination
  -> 提高相关原子的 distractor_risk
  -> 对当前角色降低其 role_affinity

proof gap
  -> 提升缺失公式、变量定义、单位、前置推导节点
  -> 下次 formal_deriver / writer payload 强制加入依赖闭包

constraint satisfied
  -> 标记 obligation covered
  -> 降低重复注入压力，但 writer 阶段仍保留必要摘要

stale intermediate output
  -> lifecycle 从 active 调整为 stale
  -> 除非 critic 或依赖闭包需要，否则不再进入常规载荷
```

这样系统从“生成-检查-修文本”升级为“生成-检查-更新上下文策略-再生成”。

### 7. 最小可行版本

第一版不需要训练模型，也不需要强化学习。建议用确定性启发式实现：

```text
score(atom) =
  obligation_link_weight
+ support_license_weight
+ dependency_centrality
+ role_affinity
+ critic_promotion
- token_cost_penalty
- distractor_risk
- stale_penalty
```

最小可行版本应完成三件事：

1. 把现有 dossier 中的重要字段映射成 typed atoms。
2. 用 OPC 替换手工拼接 payload，至少覆盖 task execution、derive、writer 和 verification。
3. 为每次 agent 调用保存 PayloadView，支持后续分析 token 成本、义务覆盖和失败原因。

### 8. 评估设计

对比基线：

```text
Full Context
  尽量塞入完整 dossier 和历史输出

Current Compact Payload
  当前手工压缩策略

Static Role Filter
  按智能体类型固定过滤字段

Similarity Retrieval
  基于关键词或 embedding 的 top-k 检索

Atom Retrieval Only
  只选相关原子，不做支持闭包和依赖闭包

OC-HMG + OPC
```

### 9. 预期贡献

该方案的贡献可以概括为四点：

1. **问题形式化**：把科学 Agent 的上下文管理定义为义务条件化的最小充分载荷选择问题，而不是普通摘要或检索问题。
2. **记忆表示**：提出带支持许可、义务链接、依赖闭包和生命周期状态的层级记忆图。
3. **载荷编译**：提出角色特定、预算感知、可解释的上下文编译器，显式优化义务覆盖、支持闭包和干扰抑制。
4. **反馈闭环**：将 verification、proof audit 和 contract failure 的结果写回记忆图，使上下文策略动态修订。

最重要的是，这个方案与普通 RAG、长上下文、prompt 压缩和静态 memory buffer 的区别非常清楚：它不是“找相关文本”，而是围绕潜在评分标准和科学声明许可来编译每次推理所需的认知载荷。
